## 1. Какво е нарушение на сигурността (Breach)?

Нарушение е всяко събитие, при което лични данни са:
- **Изтекли** — достъп от неоторизирано лице (хакер, грешка, изпратен на грешен получател)
- **Загубени** — случайно изтриване, повреда на базата данни
- **Унищожени** — данните не могат да бъдат възстановени
- **Променени** — неразрешена промяна на данни

**Примери, специфични за Nevumo:**
- Достъп до PostgreSQL база данни от неоторизирано лице
- Изтичане на JWT тайна (всички активни сесии компрометирани)
- Изпращане на данни на грешен потребител чрез API
- Достъп до Cloudflare R2 снимки без авторизация
- Компрометиране на Resend API ключ (изпращане на имейли от чуждо лице)
- SQL injection с извличане на данни

## 2. Задължителен 72-часов таймер

```
ЧАС 0:   Открито нарушение или основателно подозрение
ЧАС 4:   Вътрешна оценка — потвърждение или отхвърляне
ЧАС 24:  Ако е потвърдено → документиране в Breach Register
ЧАС 72:  КРАЕН СРОК — уведомление до КЗЛД (ако е приложимо)
```

> ⚠️ **72-те часа се броят от момента, в който Nevumo е узнала или е трябвало да узнае за нарушението — не от момента на самото нарушение.**

## 3. Стъпки при инцидент

### Стъпка 1: Установяване и ограничаване (Час 0–4)

- [ ] Идентифицирай засегнатите системи (DB, API, имейл, storage)
- [ ] Изолирай или ограничи достъпа (смени credentials, деактивирай endpoint)
- [ ] Запази доказателства — **не изтривай логове**
- [ ] Запиши точния час на установяване

**Действия по система:**

| Система | Действие при инцидент |
|---------|----------------------|
| PostgreSQL (Neon) | Смени connection string, провери audit logs, ротирай credentials |
| JWT secret | Смени `SECRET_KEY` в env → всички активни сесии се инвалидират |
| Cloudflare R2 | Смени API token, провери access logs |
| Resend | Деактивирай компрометиран API ключ, провери sent log |
| Redis (Upstash) | Ротирай token, flush ако е нужно |

---

### Стъпка 2: Оценка на риска (Час 4–24)

Отговори на следните въпроси:

**А) Вероятност от вреда за субектите:**
- Колко лица са засегнати? (< 100 / 100–1000 / > 1000)
- Какви категории данни са изтекли? (имейли / пароли / телефони / финансови)
- Има ли специални категории данни (здраве, религия, политика)? → ако да: **автоматично ВИСОК риск**
- Данните криптирани ли са? → ако да: риска е значително по-нисък

**Б) Матрица за решение:**

```
                        МАЛКО засегнати    МНОГО засегнати
Нискорискови данни      → Само регистър    → Уведомление КЗЛД
Високорискови данни     → Уведомление КЗЛД → Уведомление КЗЛД + субекти
```

### Стъпка 3: Уведомление до КЗЛД (ако е приложимо, до Час 72)
